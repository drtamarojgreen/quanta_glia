# CHAI Scenario Compiler

### Design Intent Document

## Overview

The **CHAI Scenario Compiler** is the subsystem responsible for transforming human-authored **CHAI Scenario DSL files (`.chai`)** into structured runtime artifacts used by the CHAI testing engine.

The compiler acts as a bridge between:

* **human-readable scenario definitions**
* **machine-efficient evaluation structures**

Rather than executing scenarios directly from text files, the compiler produces **validated, normalized scenario representations** that can be executed deterministically by the CHAI runtime.

The compiler ensures scenarios are:

* syntactically valid
* structurally consistent
* semantically meaningful
* optimized for execution

---

# Core Purpose

The CHAI Scenario Compiler exists to solve several problems:

### 1. Scenario Validation

Scenarios written by humans or generated by AI must be checked for:

* structural correctness
* invalid references
* logical contradictions
* unsupported configuration

The compiler acts as a **safety gate** preventing invalid tests from reaching the runtime.

---

### 2. Deterministic Scenario Execution

Raw DSL files may contain ambiguous formatting or redundant structures.

The compiler converts them into **canonical forms**, ensuring:

* consistent ordering
* normalized values
* explicit scenario structure

This allows the test engine to execute scenarios deterministically.

---

### 3. Performance Optimization

Interpreting text scenarios repeatedly during runtime would slow testing.

The compiler transforms scenarios into **efficient internal representations** that can be loaded quickly.

---

### 4. Scenario Integrity

The compiler records metadata describing:

* scenario structure
* dependencies
* evaluation rules

This ensures that scenarios remain **stable and reproducible** across test runs.

---

# Compiler Conceptual Pipeline

The CHAI Scenario Compiler processes scenario files through several conceptual phases.

```
Scenario Source (.chai)
        \u2502
        \u25bc
Lexical Analysis
        \u2502
        \u25bc
Structural Parsing
        \u2502
        \u25bc
Semantic Validation
        \u2502
        \u25bc
Normalization
        \u2502
        \u25bc
Compilation Output
```

Each phase has a specific responsibility.

---

# Phase 1 \u2014 Lexical Analysis

The lexical phase interprets raw scenario text and converts it into structured tokens.

Intent:

* identify DSL keywords
* detect string literals
* interpret scenario structure markers
* preserve meaningful ordering

This stage does **not evaluate meaning**, only syntax components.

Examples of recognized tokens include:

* scenario declarations
* section identifiers
* constraint declarations
* step definitions
* evaluation rules

The result of this phase is a **tokenized representation of the scenario**.

---

# Phase 2 \u2014 Structural Parsing

The parsing phase interprets tokens and constructs a **scenario structure tree**.

Intent:

* enforce the DSL grammar
* verify correct nesting of sections
* ensure required sections exist
* detect structural inconsistencies

For example:

* a scenario must contain exactly one name
* steps must appear inside the STEPS section
* evaluation rules must appear inside EVALUATION

The output is a **structured scenario model** representing the hierarchy of the scenario.

---

# Phase 3 \u2014 Semantic Validation

Semantic validation ensures that the scenario **makes logical sense within the CHAI testing model**.

Examples of semantic checks include:

### Environment Consistency

Ensuring environment declarations are valid.

Example:

* an operating system value must be recognized by the platform model

---

### Tool Availability

Ensuring tool definitions are meaningful.

Example:

* tools must be valid identifiers
* duplicate tool declarations are flagged

---

### Constraint Compatibility

Some constraints may conflict with others.

Example conflict:

* read-only filesystem
* write-file test requirement

The compiler detects such contradictions.

---

### Evaluation Rule Validity

Evaluation rules must correspond to supported failure classifications.

Example valid failure types:

* context drift
* tool hallucination
* platform mismatch
* constraint violation
* task oversimplification
* code navigation failure

Unsupported evaluation rules are rejected.

---

# Phase 4 \u2014 Scenario Normalization

After validation, the compiler transforms the scenario into a **normalized representation**.

Normalization ensures that:

* data ordering is consistent
* redundant elements are removed
* default values are filled in

Examples of normalization behavior:

### Canonical Ordering

Sections may be reordered internally into a consistent structure.

---

### Default Environment Values

If an environment property is unspecified, the compiler may assign defaults.

Example:

* architecture defaults to x86_64

---

### Constraint Expansion

Some constraints may expand into multiple internal checks.

Example:

```
no_internet
```

May internally expand to restrictions on:

* network calls
* remote downloads
* API queries

---

# Phase 5 \u2014 Compilation Output

The final output of the compiler is a **compiled scenario artifact**.

The artifact contains:

* scenario metadata
* normalized environment description
* tool registry
* constraints
* message sequences
* evaluation configuration

The output is optimized for direct loading by the CHAI runtime engine.

---

# Output Artifact Goals

Compiled scenarios should be:

* lightweight
* deterministic
* easy to load
* version controlled

The format should preserve all scenario information while eliminating ambiguity.

---

# Scenario Identity and Versioning

The compiler assigns each scenario a **unique scenario identity**.

Intent:

* ensure reproducibility
* allow scenario comparisons
* support scenario evolution

Scenario metadata may include:

* scenario name
* compilation timestamp
* scenario version
* DSL version used

This allows scenarios to remain stable even as the DSL evolves.

---

# Scenario Diagnostics

When errors occur, the compiler provides detailed diagnostics.

Intent:

* help scenario authors correct mistakes
* enable automated generation systems to adjust scenarios

Diagnostics may include:

* syntax errors
* structural errors
* semantic conflicts
* unsupported evaluation rules

Clear diagnostics improve **developer and AI-generated scenario quality**.

---

# Scenario Integrity Checks

The compiler ensures scenarios cannot silently change behavior.

Integrity checks may include:

* scenario fingerprinting
* structure hashing
* evaluation rule verification

This prevents subtle corruption or unintended modifications.

---

# Interaction With CHAI Runtime

The CHAI runtime interacts only with **compiled scenarios**, not raw DSL files.

This separation provides several advantages:

* runtime performance
* stability
* easier debugging
* predictable behavior

The runtime focuses exclusively on:

* scenario execution
* agent interaction
* failure detection
* metrics collection

---

# Scenario Compilation Workflow

Typical workflow:

1. Scenario author writes `.chai` file.
2. CHAI Scenario Compiler validates the file.
3. Scenario is compiled into a runtime artifact.
4. Runtime engine loads compiled scenario.
5. AI agents are evaluated using the scenario.

This separation mirrors traditional software compilation pipelines.

---

# Support for Generated Scenarios

One of the most powerful design goals is enabling **automated scenario generation**.

AI systems may produce thousands of scenario variations.

The compiler ensures that generated scenarios are:

* structurally valid
* logically consistent
* usable by the runtime engine

Thus the compiler acts as a **quality filter for generated tests**.

# Design Philosophy

The CHAI Scenario Compiler follows several guiding principles.

### Determinism

Scenario behavior must remain consistent across runs.

---

### Transparency

Scenario structure must remain understandable to both humans and machines.

---

### Extensibility

New failure modes and evaluation rules must be easy to add.

---

### Agent Compatibility

Scenario structures should remain readable and interpretable by AI systems themselves.

---

# Summary

The CHAI Scenario Compiler serves as the **validation and transformation layer** between the CHAI Scenario DSL and the runtime evaluation engine.

Its intent is to ensure that scenarios are:

* structurally correct
* logically coherent
* execution-ready
* reproducible

By separating **scenario authoring from scenario execution**, the compiler enables CHAI to scale to **large libraries of AI evaluation tests**, while maintaining reliability and clarity.
